Onboarding & DX feedback log

Captured live during the build — friction and delight, timestamped. This is the as-it-happened record; the triaged, presentation-ready version is quick-fixes.html.

⚠️🔴 friction🟡 rough edge✅🟢 delight🧭 theme

2026-06-09 — setup

  • ⚠️SDK drift is real. Agent(instrument=True) (in many tutorials / our own REFERENCES.md) is deprecated in pydantic-ai 1.x — the supported switch is Agent.instrument_all(True). Had to read installed source to find it. Onboarding implication: Arize tracing snippets pinned to a framework version go stale silently; a "verify against your installed version" nudge (or a version-aware quickstart) would save a confusing first-trace miss.
  • ✅arize.otel.register(span_processors=[...]) cleanly accepts the OpenInference processor, so ordering vs the exporter is handled for you. Phoenix's register() lacks that arg → you must .add_span_processor() after, an easy ordering footgun. The asymmetry between the two register()s is a small DX paper-cut worth noting.
  • ✅One OpenRouter key for Claude/GPT/Gemini makes the "provider-neutral" story trivial to set up.
  • 🟡Python 3.14 (system) risked dependency wheels; pinning the venv to 3.12 via uv avoided it. A docs note on supported Python versions would preempt this.

2026-06-09 — Phase 3 (dataset + experiment + evals)

  • 🔴Biggest footgun: experiment tasks must be async. A sync task using agent.run_sync() silently fails inside the experiment's running event loop (asyncio.run() cannot be called from a running loop) → every task output is None → ALL eval scores come back 0.0 with no obvious error in the summary. The fix (async def task + await agent.run()) isn't surfaced in the quickstart; the failure mode looks like "my evaluators are broken," not "my task threw." A first-class diagnostic — "N tasks errored, see traces" in the run summary, or a sync-task autodetect — would save hours.
  • 🟡SDK API surface is non-obvious and partly BETA: datasets.get(dataset=<name>) (not name=), space passed per-call (not on the client), uploaded columns land under additional_properties. Introspecting installed source was faster than the docs.
  • 🟡Dataset name collisions throw a raw 409 ConflictException — a get-or-create helper or create(..., exist_ok=True) would be friendlier.
  • 🟢Once tasks were async, the dev workflow is genuinely tight: one experiments.run call gives per-row outputs + multi-judge scores, and dry_run is a real safety net.
  • 🟢Ingest ARIZE_API_KEY authorized BOTH tracing and the experiments SDK — no separate developer key needed (a pleasant surprise; AX docs imply otherwise).

2026-06-09 — Phase 4 (deliberate failure / observability)

  • 🟢Holding the judge model constant while swapping the agent model (one OpenRouter key) made the 3-provider comparison trivial — this is the single most persuasive Arize demo and it cost ~2 commands. Worth featuring in onboarding.
  • 🟢A degrade knob in the task (drop a corpus doc) + re-running the SAME experiment gives a clean before/after the eval columns capture automatically. The story writes itself: score drop → low-score traces → empty/weak RETRIEVER span → root cause → fix → recovery.
  • 🟡Online-eval Tasks are an AX-UI surface; wiring one from the SDK wasn't obvious (arize.tasks/arize.evaluators clients exist but undocumented for this). For the demo, the offline experiment delta tells the same story; an online Task on the live project is the remaining UI step to capture.

2026-06-11 — trace-review UX (observability workflow, in-product)

Friction found while actually reading traces in the AX UI — the core day-2 loop, distinct from setup. Theme: the trace detail view isn't a single pane of glass.

  • 🟡Hidden horizontal scroll on the detail panes. Switching between views in a span's detail (Input/Output vs. error/exception vs. other attributes), the content is side-scrollable but gives no affordance that it is — no fade edge, scrollbar, or "→ more" cue. It's easy to read a pane as complete and miss detail parked off-screen to the right. Product implication: long values (prompts, tool args, retrieved docs) should wrap/expand by default, or carry an explicit overflow affordance. Silent horizontal overflow in a debugging tool actively hides the thing you opened the trace to see.
  • 🔴Tool-call-only spans read as "no output." When a span's execution is purely a tool call, the main Output area shows nothing — the actual result is only visible under Attributes. The default view implies the step produced nothing, which is misleading and breaks the scan-down-the-waterfall flow. Product implication: the span summary should surface the tool result (name + args + return) inline in the primary pane. For tool/retriever spans the OpenInference attributes ARE the output — the UI should promote them.
  • 🧭Root theme → single pane of glass. Both issues are the same gap: following one trace end-to-end requires hunting across tabs, panes, and an off-screen axis. The ergonomic target is reading the full plan→tool→result→eval story top-to-bottom without clicking into Attributes or scrolling sideways. A clean onboarding/observability-DX recommendation and a candidate talking point: the trace view should make the next thing you need to see the thing that's already on screen.

2026-06-12 — evaluator lifecycle / production eval quality

Found while reading the AX Evaluators + annotation docs — the day-2 question of how you keep an eval honest AFTER it's live. Both items below are candidate product concepts (seeded as H/I in the concept ledger).

  • 🟡No live eval↔human divergence view. Annotations and LLM-judge evals both land on the same span, but nothing surfaces where they disagree on production data — you detect judge drift by rebuilding an offline golden set and re-running agreement metrics, not in situ. Product implication: a first-class "judge vs. human" divergence monitor on live spans would close the loop annotations open.
  • 🟡No inter-annotator agreement / labeler-noise signal. Labeling queues allow multiple assignees but expose no κ/α or consensus metric — so when an eval won't converge you can't tell labeler noise from a weak judge prompt. (Red-team precision check: AX does ship "Align Evals to Human Feedback" — a confusion matrix comparing one judge vs. human on a calibration set. What it does not ship is human-vs-human IAA: queues dedupe each item to a single reviewer. So Align Evals assumes clean human labels; it never measures whether the humans agree.) Product implication: add N-way overlap to labeling queues + a per-item agreement/κ panel that gates Align Evals, plus an "this evaluator is overloaded, split it into atomic criteria" diagnostic.
  • 🧭Theme → trust the humans and the rubric, not just the judge. These extend the same eval-quality arc as concept G: G asks "is a generated eval good before you ship it?"; these ask "are your human labels consistent, is your live judge still agreeing with them, and is your rubric trying to do too much?" (Disposition: folded into concept G — Langfuse shipped the live human-vs-judge monitor and IAA is commoditized in Label Studio/Argilla. See concept-ledger.html cards H/I.)

2026-06-13 — experiment vs. production eval review (verified with Alyx)

Found while reviewing failed experiment cases; confirmed live with Arize's in-product assistant (Alyx). Logged as Quick Fixes #8 and #9.

  • 🟡Reviewing an experiment's evals is a different pattern than production's. Production traces show evals on the Evals tab of the trace view. Experiment-run verdicts live on the experiment run record (the red/green labels in the compare table) and are never written onto the trace spans — so that tab is empty when you open a trace reached from an experiment. Debugging a failed row means bouncing between the compare table (the verdict) and the trace (the evidence). Confirmed deliberate with Alyx: "Experiment eval scores are stored on the experiment run record, not written back onto the trace spans… most users use the compare table as the primary place to review." That intent is the problem — it optimizes for the table author, not the trace debugger. Product implication: surface experiment eval verdicts on the Evals tab of the trace review UI too — same single-pane-of-glass theme as the 6/11 items.
  • 🟡Alyx can't export its own transcript. "I don't have a tool to save this to a file… that's a limitation of the current chat interface" — only copy/paste, screenshot, or print-to-PDF. This bit precisely because the exchange was worth keeping: users mine these conversations and carry them into other tools. Product implication: one-click Export/Copy transcript (markdown or PDF) on the Alyx panel turns answers into reusable artifacts. Quick Fix #9.
