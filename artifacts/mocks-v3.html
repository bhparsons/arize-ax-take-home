From Signal to Decisions

Wireframes for the proposed Signal → Decisions surfaces — a demo concept, not final UI. They follow one DoorDash support agent through a single turn of the loop: set the objective, see issues ranked by business value, then fork down two paths — a policy decision (A) and a shared-capability gap (B).

Screen 1 Setup — the objective that ranks every decision

This is what ranks everything in the later screens. Evals keep running on their own — the objective just decides which eval failures are worth acting on.

Signal / Decisions / #204 / the issue
A per-window completion quota caps the agent — correct sessions fall back to a human
change: Policyowner: policy team1,240 sessions · 5.0% of volumetrend ▲ 6 wks
Trigger · escalation-necessity eval

Would a reasonable agent have needed to hand this off? Flags sessions where reasoning was correct but blocked — a class correctness evals pass and outcome data exposes.

Root cause · Alyx

A per-window quota cap on order-completions, not a model error. Reasoning correct in 94% of the cluster; failing span is mark_delivered → quota_exceeded → forced human fallback.

Evidence · 1,240 traces (Dashers asking to complete their own delivery)
  • session 8f21 · "mark my order done"capped → human fallback
  • session a04c · "I delivered it already"capped → human fallback
  • session 7be9 · "complete order #5512"capped → human fallback
  • + 1,237 more in this clusterview traces ›

The issue isn't bad reasoning — it's a cap. The escalation-necessity eval surfaces a class of failure correctness evals miss: the agent did the right thing and was still blocked, because an order-completion quota limits how many deliveries it may mark complete per window. The evidence pulls the traces that prove it. Next: what the platform does about a control it can't adjudicate. → Screen 4.

The completion quota caps 1,240 correctly-reasoned sessions into human fallback. The platform can't adjudicate a fraud control — so it routes the evidence with a concrete ask: raise the cap for verified, clean-history Dashers and keep the quota for high-risk users. That conservatively captures the lift.
ceiling +3.1pt · 5% vol × ~62% recoverable constraint · erroneous-payout rate held — quota kept for high-risk evidence: 1,240 traces · 94% correct reasoning
Route to policy owner → share link Export artifact (PDF) Decline with reason
If declined — the objective listens

A decline with reason (e.g., "keep the quota — it's a fraud control") proposes a new constraint on the objective — visible on Screen 1. Deviation isn't noise; it's how the objective gets truer. Declined-with-reason is a successful terminal state, not a failure.

The CTA matches the change type. A policy decision routes an evidence pack to its owner — it doesn't "open a PR" (4 of 6 change types don't live in the repo). The status timeline makes the decision, not the recommendation, the unit of progress. And the recommendation is conservative by construction: raise the cap only where risk is low, keep the control where it isn't.

The aggregation is the insight — and it's observability-native. Each issue sits below the threshold to justify a build; only summed do they clear it — and only the platform can see the sum, because each issue lands in a different owner's queue. The same pattern fits a GPS-trace evaluator (geo disputes, wrong-address, distance-fraud — one capability, many intents). Next: the spec. → Screen 7.

tool: evaluate_delivery_photo # draft — generated from Issue #218, edited by [owner] purpose: let the agent look at a submitted photo and answer what each photo intent needs, so image-gated cases resolve without a blind escalation trigger_conditions: - intent ∈ {delivery_dispute, damaged_item, wrong_address_claim, …} # 14 intents - an image attachment is present on the session modes: # what the tool is asked to look for, per intent match_prior_delivery: compare photo ↔ past successful deliveries for this address → {match, confidence} inspect_contents: identify order size · claimed items · visible damage → {items_present, damage, confidence} # extensible: a new photo intent adds a mode, not a new tool inputs: image (customer_upload) · order_id (session context) · mode (intent → mode) evidence: 412 sessions escalated with images acknowledged but unevaluated (traces ›) projection: ceiling +2.2pt resolution rate across all 14 intents constraint_exposure: erroneous payouts — medium → auto-resolve only at confidence ≥0.8, else current escalation path open_questions_for_eng: # the platform drafts — the owner finishes - reuse an existing CV service, or a net-new model per mode? - PII handling for customer-uploaded images — retention window?
Send spec to engineering Edit spec View evidence traces

The spec is collaborative by design. Engineering receives a scoped, evidenced, prioritized draft — naming what each mode looks for, not "the agent should handle images." The cross-intent footprint is the value: one roadmap item worth +2.2pt, with the open questions left for the owner to finish. (V2 extends this rollup across the whole agent fleet.)
